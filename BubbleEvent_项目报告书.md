## 1. 项目概述与建设目标

### 1.1 项目背景

在当今信息爆炸的金融市场中，热点事件以碎片化形式分散于新闻、财报、社交媒体、研报等多种信息渠道中。投资机构与研究者面临以下核心痛点：

| 痛点 | 具体表现 |
|------|----------|
| **信息过载** | 每日海量新闻与社交信息，人工筛选成本极高 |
| **真伪难辨** | 谣言、误传与真实信息混杂，缺乏系统化验证手段 |
| **映射困难** | 事件到资产的传导链缺乏结构化分析框架 |
| **反映速度** | 市场往往提前反映预期，人工难以实时跟踪 |
| **经验孤岛** | 投资经验依赖个人，缺乏系统化积累与复用机制 |

**核心理念**：市场对信息的反映不是终点，而是新一轮认知的起点。事件驱动的本质不是"追涨杀跌"，而是"发现→验证→学习"的持续迭代过程。

### 1.2 建设目标

#### 业务目标

> 回答三个核心问题：**发生了什么？是否真实？是否已经被市场定价？**

- **广度覆盖**：全渠道信息源采集，不遗漏重大事件
- **深度验证**：多维度交叉验证信息可信度，过滤噪音
- **快速映射**：事件到受影响资产/行业的自动关联
- **持续追踪**：从事件爆发到市场消化全周期跟踪

#### 产品目标

- **事件卡片**：结构化呈现事件全貌（摘要、等级、可信度、影响资产、风险提示）
- **复盘卡片**：T+1/T+3/T+7/T+30 多时间窗口自动复盘
- **日报/周报**：每日快报和每周深度复盘，支持用户自定义订阅行业/资产
- **API 服务**：为下游量化策略和研究平台提供结构化事件数据接口

#### 系统目标

> 形成"发现 → 验证 → 因果判断 → 追踪 → 复盘 → 规则更新"的学习闭环

### 2.2 数据流闭环

```
原始信息源 → 采集层 → 事件提取 → 聚类去重 → 可信度验证
                                                    ↓
用户交付 ← 卡片生成 ← 市场验证 ← 资产映射 ← 影响分析
    ↓                      ↓
复盘反馈 ←──────────────── 预测账本
    ↓
权重调整 → 规则更新 → 知识库沉淀 → 提升下一次分析质量
```

### 2.3 核心设计原则

| 原则 | 说明 |
|------|------|
| **Agent 专职化** | 每个 Agent 只负责一个明确子任务，降低耦合，便于独立优化 |
| **管道式处理** | 上游输出作为下游输入，每阶段可独立验证与调试 |
| **可解释性** | 每个分析结论附带推理链，支持人工追溯和复核 |
| **持续学习** | 预测与实际对比结果自动反馈，驱动权重和规则持续进化 |
| **人工在环** | 关键节点保留人工确认入口，不追求全自动黑箱决策 |

## 3. 多 Agent 协作体系设计

### 3.1 Agent 总览

系统共设计 **12 个核心 Agent** + **4 个扩展 Agent**，按职责分为五个层级：

| 层级 | Agent 名称 | 职责 | 输入 | 输出 |
|------|-----------|------|------|------|
| **采集层** | ① Information Scout Agent | 全渠道信息采集 | 配置的信息源列表 | 原始信息流 |
| **理解层** | ② Event Extraction Agent | 事件提取与结构化 | 原始信息流 | 结构化事件对象 |
| | ③ Event Clustering Agent | 事件去重、聚类、合并 | 结构化事件列表 | 去重后的事件簇 |
| **验证层** | ④ Credibility Verification Agent | 信息真伪验证 | 事件簇 | 带可信度标签的事件 |
| | ⑤ Event Scoring Agent | 事件重要性评级 | 已验证事件 | 带优先级评分的事件 |
| **分析层** | ⑥ Causal Reasoning Agent | 因果推理与市场影响分析 | 已评分事件 | 因果链 + 影响判断 |
| | ⑦ Historical Analogy Agent | 历史相似事件类比 | 事件 + 历史数据库 | 历史对标分析 |
| | ⑧ Anti-Spurious Reasoning Agent | 反伪相关推理 | 因果链 + 市场数据 | 排除伪相关的净化结论 |
| | ⑨ Market Mapping Agent | 资产/行业/公司映射 | 分析结论 | 受影响资产清单 |
| **验证-** | ⑩ Market Data Verification Agent | 实时行情验证 | 资产清单 + 行情数据 | 市场确认信号 |
| **交付层** | ⑪ Card Generation Agent | 事件卡片与报告生成 | 全链路数据 | 事件卡片/日报/API |
| **进化层** | ⑫ Review & Learning Agent | 复盘、学习与权重更新 | 预测账本 + 实际数据 | 规则更新 + 知识沉淀 |

#### 扩展 Agent

## 4. 事件发现与验证管道

### 4.1 信息采集层（Information Scout Agent）

#### 信息源覆盖范围

| 来源类型 | 具体渠道 | 采集方式 |
|----------|---------|----------|
| 官方新闻 | 新华社、证监会、央行、各部委官网 | RSS / API |
| 财经媒体 | 财联社、华尔街见闻、 Bloomberg、Reuters | RSS / Scrapy |
| 社交媒体 | 微信公众号、微博、Twitter/X、Reddit | Playwright / API |
| 研报平台 | Wind、东方财富、分析师报告 | API / 爬虫 |
| 行情数据 | A股/港股/美股实时行情、期货、汇率 | Wind / Tushare / FRED |
| 公告披露 | 巨潮资讯、SEC EDGAR、港交所披露易 | API / 爬虫 |
| 行业垂直 | 各行业专业媒体、行业协会公告 | RSS / 爬虫 |

#### 采集策略

- **常规轮询**：每 1-2 小时对全量信息源进行扫描
- **事件驱动加速**：检测到高重要性事件时，自动提高信息源轮询密度
- **关键词过滤**：支持用户自定义关注的行业/资产/关键词集合

### 4.2 事件提取与结构化（Event Extraction Agent）

从非结构化文本中提取关键要素，输出标准化 JSON 事件对象：

```
输入：“美国商务部宣布对中国电动汽车加征100%关税，将于2026年8月1日生效”

输出：
{
  "event_id": "EVT_20260615_001",
  "event_type": "trade_tariff",
  "event_title": "美国对中国电动汽车加征100%关税",
  "entities": ["美国商务部", "中国电动汽车产业"],
  "event_category": "政策/关税",
  "timestamp": "2026-06-15T10:30:00Z",
  "effective_date": "2026-08-01",
  "location": "美国",
  "affected_industries": ["新能源汽车", "锂电池", "汽车零部件"],
  "raw_sources": ["Reuters", "Bloomberg"],
  "confidence": 0.95
}
```

#### 提取要素清单

| 要素 | 说明 | 示例 |
|------|------|------|
| 事件主体 | 涉及的公司/组织/个人 | "美国商务部" |
| 事件类型 | 分类标签 | "贸易关税/自然灾害/财报/并购/..." |
| 时间 | 事发时间和生效时间 | "2026-06-15 / 2026-08-01" |
| 地点 | 事件发生地/影响市场 | "美国/全球" |
| 影响行业 | 受影响的产业标签 | "新能源汽车/锂电池" |
| 信息来源 | 原始信息链接 | "Reuters URL" |

### 4.3 事件聚类与去重（Event Clustering Agent）

#### 处理逻辑

1. **语义去重**：基于事件主体、类型、时间的向量相似度计算，合并重复报道
2. **更新识别**：区分"首发报道"与"后续进展"，维护事件时间线
3. **关联发现**：识别不同来源对同一事件的多角度报道，合并为完整事件画像
4. **不确定性标记**：对信息不完整的事件标注 `uncertainty: true`，降低下游分析权重

#### 输出示例

```
事件簇 #CLUSTER_042:
  主事件: "中美芯片出口管制新规"
  子事件:
    - [06-15 09:00] 路透社首次报道 → 首发
    - [06-15 11:30] 彭博补充细节 → 更新
    - [06-15 14:00] 中国商务部回应 → 关联
  去重后输出: 1 个合并事件 + 2 条时间线补充
```

### 4.4 可信度验证（Credibility Verification Agent）

#### 五维验证框架

| 验证维度 | 权重 | 验证方法 | 评分标准 |
|----------|------|----------|----------|
| **来源等级** | 25% | 信源权威性分级数据库 | 官方公告 1.0 → 头部媒体 0.8 → 自媒体 0.3 |
| **来源确认** | 20% | 多独立信源交叉比对 | ≥3 独立来源确认：高分 |
| **官方确认** | 25% | 官方渠道（政府/公司/交易所）核实 | 官方确认 1.0；未确认 0.0 |
| **时间一致性** | 15% | 各来源时间戳是否一致 | 多源时间一致：高分 |
| **历史准确率** | 15% | 该信源历史报道准确率统计 | 基于历史复盘数据动态计算 |

#### 输出示例

```
{
  "event_id": "EVT_20260615_001",
  "credibility_score": 0.91,
  "verification_status": "confirmed",
  "evidence_chain": [
    {"source": "White House Official Statement", "type": "primary"},
    {"source": "Reuters", "type": "secondary"},
    {"source": "Bloomberg", "type": "secondary"}
  ],
  "credibility_flags": []
}
```

#### 处置分级

| 等级 | 可信度区间 | 处置策略 |
|------|-----------|----------|
| **S 级** | ≥0.85 | 高优先级推送，触发全链路分析 |
| **A 级** | 0.70-0.84 | 重点跟踪，待进一步确认后升级 |
| **B 级** | 0.50-0.69 | 低频跟踪，标注"待确认" |
| **C 级** | <0.50 | 静默观察，不主动推送 |
| **D 级** | 确定为假 | 归档标记，用于信源可信度下调 |

## 5. 市场影响分析管道

### 5.1 事件重要性评分（Event Scoring Agent）

综合多维度对事件进行重要性评级，决定分析资源的分配优先级：

| 评分维度 | 权重 | 说明 |
|----------|------|------|
| 市场相关性 | 30% | 事件与资本市场的直接关联度 |
| 影响范围 | 25% | 影响的行业广度、资产类别数量 |
| 影响深度 | 25% | 对单一行业/公司的冲击程度 |
| 可解释性 | 10% | 因果链条是否清晰可追踪 |
| 时效性 | 10% | 信息的新鲜度和决策窗口期 |

### 5.2 因果推理（Causal Reasoning Agent）

构建从事件到市场影响的因果传导链：

```
事件: 美国对中国电动汽车加征100%关税
  │
  ├─→ 直接影响: 中国电动汽车对美出口下降
  │     └─→ 比亚迪/蔚来等出口型企业营收承压
  │
  ├─→ 衍生影响: 锂电池需求预期下调
  │     └─→ 宁德时代/亿纬锂能等供应链企业估值压力
  │
  ├─→ 间接影响: 中国内需刺激政策预期升温
  │     └─→ 国内消费/基建板块可能受益
  │
  └─→ 宏观影响: 中美贸易摩擦升级预期
        └─→ 避险情绪 → 黄金/国债
```

#### 因果链输出结构

```
{
  "event_id": "EVT_20260615_001",
  "causal_chains": [
    {
      "chain_id": "CC_001",
      "direction": "negative",
      "target_assets": ["比亚迪", "蔚来", "宁德时代"],
      "transmission_logic": "出口成本上升 → 销量下降 → 营收承压 → 估值下调",
      "confidence": 0.82,
      "time_horizon": "1-3个月"
    }
  ]
}
```

### 5.3 历史类比（Historical Analogy Agent）

从历史事件库中检索相似事件，对比分析：

| 对比维度 | 当前事件 | 历史对标事件 | 相似度 |
|----------|---------|-------------|--------|
| 事件类型 | 关税加征 | 2018年中美贸易摩擦 | 0.78 |
| 影响规模 | 电动汽车单一行业 | 多行业关税 | 0.65 |
| 市场环境 | 牛市中期 | 震荡市 | 0.42 |
| **综合相似度** | | | **0.62** |

历史对标输出：**2018年贸易摩擦期间，相关出口型股票在事件后1个月平均下跌 8-15%，3个月后分化——内需替代型标的反弹显著。**

### 5.4 反伪相关推理（Anti-Spurious Reasoning Agent）

主动识别和排除伪相关性，防止把巧合当因果：

| 伪相关类型 | 示例 | 排除方法 |
|-----------|------|----------|
| 时间巧合 | "某事件发生同时某股票跌了" | 检验历史同期波动率 + 统计显著性 |
| 第三方变量 | "降息 + 股市涨 + 事件X" → 实际是降息驱动 | 控制变量法，隔离宏观因子 |
| 反向因果 | "股票跌了所以出了负面新闻" | 时序检验：新闻时间 vs 股价异动时间 |
| 幸存者偏差 | 只关注涨了的股票与正面事件 | 构建完整对照样本 |

### 5.5 资产映射（Market Mapping Agent）

将事件影响传导到具体可交易资产：

```
事件 → 直接影响行业 → 行业中的上市公司/ETF/期货合约

示例：
  "AI芯片出口管制"
    ├─→ AI芯片设计: 寒武纪、海光信息
    ├─→ 先进封装: 长电科技、通富微电
    ├─→ EDA工具: 华大九天、概伦电子
    ├─→ 半导体设备: 中微公司、北方华创
    ├─→ ETF: 芯片ETF(159995)、科创50ETF(588000)
    └─→ 期货: 中证500股指期货
```

### 5.6 市场数据验证（Market Data Verification Agent）

验证分析结论是否已被市场定价：

| 验证维度 | 检测方法 | 验证信号 |
|----------|---------|----------|
| **价格异动** | 事件前后窗口期收益率对比 | 是否出现超常波动 |
| **成交量** | 成交量相对于历史均值的偏离 | 是否放量（确认信号） |
| **资金流向** | 北向资金/主力资金净流入 | 聪明钱方向 |
| **隐含波动率** | 期权隐含波动率变化 | 市场定价的不确定性 |
| **跨市场验证** | 相关资产是否一致反映 | 逻辑链是否在多资产间成立 |

#### 市场确认分级

```
{
  "event_id": "EVT_20260615_001",
  "market_confirmation": "partial",
  "confirmations": [
    {"asset": "比亚迪", "signal": "price_drop_3%", "confirmed": true},
    {"asset": "宁德时代", "signal": "price_drop_2.4%", "confirmed": true},
    {"asset": "芯片ETF", "signal": "no_significant_move", "confirmed": false}
  ],
  "overall_assessment": "部分确认——直接相关资产已反应，衍生资产尚未定价"
}
```

确认等级：
- **full**: 多数映射资产出现一致异动 → 市场已反映
- **partial**: 部分资产异动，部分尚未反应 → 存在预期差
- **none**: 无明显异动 → 市场尚未定价或逻辑错误
- **contrary**: 出现反向异动 → 需重新评估因果链

## 6. 学习复盘闭环机制

> **复盘不是附加功能，而是系统持续进化的核心引擎。**

### 6.1 预测账本（Prediction Ledger）

每次事件分析完成后，自动写入预测账本，记录关键字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| `event_id` | 事件唯一标识 | EVT_20260615_001 |
| `event_level` | 事件等级 | S / A / B / C |
| `credibility_score` | 可信度评分 | 0.91 |
| `impact_score` | 影响评分 | 0.78 |
| `causal_chains` | 因果链记录 | [CC_001, CC_002, ...] |
| `predicted_assets` | 预测受影响资产 | ["比亚迪", "宁德时代"] |
| `direction` | 预测方向 | negative / positive / neutral |
| `time_window` | 预测时间窗口 | T+1 / T+3 / T+7 / T+30 |
| `confidence` | 预测置信度 | 0.82 |
| `risk_flags` | 风险标记 | ["政策风险", "执行不确定性"] |
| `review_schedule` | 复盘时间表 | [1d, 3d, 7d, 30d, 90d] |

### 6.2 自动复盘调度

| 复盘节点 | 复盘重点 | 触发条件 |
|----------|---------|----------|
| **T+1 日** | 市场短期反应：方向是否正确？幅度是否匹配？ | 自动触发 |
| **T+3 日** | 市场消化程度：是否出现均值回归？新信息补充？ | 自动触发 |
| **T+7 日** | 中期验证：因果链各环节是否成立？ | 自动触发 |
| **T+30 日** | 长期影响：预期是否过度/不足？ | 自动触发 |
| **T+90 日** | 大周期验证：趋势性判断是否正确？ | 可选触发 |

### 6.3 复盘验证维度（六维验证）

```
            ┌─────────────────┐
            │   复盘验证维度    │
            └────────┬────────┘
                     │
    ┌─────┬─────┬────┼────┬─────┬─────┐
    ▼     ▼     ▼    │    ▼     ▼     ▼
  方向   幅度   时序  │  资产   逻辑   信源
  验证   验证   验证  │  验证   验证   验证
                     │
  预测涨/跌  预测涨  事件→市场  映射的  因果链   信息源
  方向是否  跌幅是  反应的时  资产是  逻辑是  事后是
  正确？   否匹配？ 间差是否  否确实  否经得  否被证
                   符合预期？ 异动？  起检验？ 伪？
```

### 6.4 权重自动更新机制

基于复盘结果，自动调整各 Agent 和知识库中的权重参数：

| 更新对象 | 更新依据 | 更新方式 |
|----------|---------|----------|
| 信源可信度权重 | 该信源历史准确率统计 | Bayesian 更新 |
| 因果规则强度 | 因果链预测准确率 | 强化学习（正确→加固，错误→衰减） |
| Agent 输出置信度 | 各 Agent 预测结果的回溯验证 | 滑动窗口统计 |
| 资产映射关系 | 实际异动资产 vs 映射资产 | 召回率/准确率优化 |
| 个性化阈值 | 用户反馈行为（点击/忽略/纠正） | 在线学习 |

### 6.5 复盘结果处置

```
复盘结论 ─┬─→ 预测正确 → 强化因果规则权重，记录为成功案例
          │
          ├─→ 方向正确但幅度有偏差 → 调整影响量化参数
          │
          ├─→ 方向错误 → 标记因果链失败，降低相关规则权重
          │        └─→ 检查是否伪相关、遗漏变量、或市场异常
          │
          └─→ 逻辑正确但市场未反应 → 标注"前置判断"，延长追踪窗口
```

## 8. 知识库体系设计

### 8.1 知识库分类

## 12. 合规与风险提示

### 12.1 合规设计原则

| 原则 | 实现方式 |
|------|---------|
| **非投资建议** | 所有输出明确标注"仅供参考，不构成投资建议" |
| **可追溯性** | 每个分析结论附带原始信息来源链接 |
| **不确定性披露** | 可信度评分、置信度区间与风险提示同步展示 |
| **不主动荐股** | 输出影响分析而非买卖建议，不给出目标价、仓位、杠杆 |
| **合规审查 Agent** | 独立 Agent 审核输出内容，过滤违规表述 |

### 12.2 内容安全边界

- 明确禁止：给出精确目标价、建议杠杆率、保证收益承诺、煽动性交易建议
- 允许范围：事件影响逻辑分析、历史对标参考、市场反应情况描述、风险因素提示

### 12.3 免责声明模板

> 本系统输出的所有内容仅为事件研究与市场分析参考，不构成任何形式的投资建议。市场价格的变动受多种因素影响，可能提前反映或过度反映相关信息。投资决策应结合个人风险承受能力，并在专业顾问指导下做出。

## 13. 总结与愿景

### 13.1 一句话总结

> **BubbleEvent 是一个"会复盘的"事件驱动投资研究辅助系统——它不只告诉你发生了什么，还会追踪这件事后来怎么样了、当初的判断对不对、以及从中能学到什么。**

### 13.2 核心差异点

| 维度 | 传统信息终端 | BubbleEvent |
|------|------------|------------|
| 信息处理 | 人工筛选 | Agent 自动采集+提取+去重 |
| 真伪验证 | 人工判断 | 五维自动验证 + 信源可信度库 |
| 影响分析 | 经验驱动 | 因果推理 + 历史类比 + 反伪相关 |
| 市场验证 | 事后回顾 | 实时行情交叉验证 |
| 学习进化 | 个人积累 | 预测账本 + 自动复盘 + 权重更新 |
| 可解释性 | 结论导向 | 因果链透明 + 全链路可追溯 |

### 13.3 长远愿景

- **知识复利**：每一次事件的追踪→复盘→学习，沉淀为系统知识，使分析能力随时间指数增长
- **认知外脑**：不仅是信息聚合工具，更是具备"事件思维"的认知协作伙伴
- **生态连接**：通过 API 和开放知识库，连接量化策略、研究平台和投资决策流程

---

> **设计哲学**：市场对信息的反映不是终点，而是新一轮认知的起点。BubbleEvent 不仅关注"发生了什么"，更关注"市场是否正确反映了它"以及"我们从中能学到什么"。这是一个持续进化的系统——每一次判断、每一次复盘、每一次修正，都在让它变得更聪明。
